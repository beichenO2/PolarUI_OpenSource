# Native Workflow Web Domain Persistence Design

**Status:** Approved by the existing native-template design and the instruction to complete Phase 3 without further checkpoints on 2026-07-15.

**Phase:** Native Workflow Web Template Phase 3.

**Scope owner:** `PolarUI/templates/native-web` and the native release verification and production QA pipeline.

## 1. Goal

Phase 3 replaces the Phase 1 placeholder workspace with persisted, user-owned workflow navigation:

`User -> Context -> Route -> Stage -> Thread -> Checkpoint`

The browser must let an authenticated user create and reopen Contexts, browse every manifest Stage without changing workflow state, create multiple Stage Threads, inspect immutable Checkpoints, and derive a new Route from a historical Checkpoint. A refresh or Web-container replacement must restore the same server data and URL-selected workspace.

## 2. Boundary

Phase 3 implements domain persistence, ownership, HTTP APIs, and the corresponding Web UI. It does not call PolarFlow, stream workflow output, store messages or attachments, adopt Thread conclusions, or execute manifest named actions. Those remain Phase 4 and later work.

Native stays opt-in and the legacy export flavor remains the default.

## 3. Domain invariants

1. Every Context belongs to exactly one user.
2. Every Route belongs to one Context and has one immutable origin Checkpoint reference when branched.
3. Checkpoints are append-only snapshots. PostgreSQL rejects update and delete operations.
4. A Route head may only move to a Checkpoint owned by the same Route.
5. Stage definitions come from the release manifest; Stage projection rows may reference only keys present in that manifest.
6. Browsing a Stage or Checkpoint never changes the Route head.
7. A Thread is scoped to one Context, Route, and Stage. Creating a Thread does not create a Checkpoint.
8. Branching from any visible Checkpoint creates a new Route with a cloned snapshot and leaves the source Route unchanged.
9. Every repository query includes the authenticated user scope. Cross-user identifiers return not-found responses rather than revealing ownership.
10. Context, Route, Thread, and Checkpoint identifiers are UUIDs generated by the application.

## 4. PostgreSQL model

Migration `0002_workflow_domain.sql` adds:

- `contexts`: user-owned problem spaces with title and active/archived status;
- `workflow_routes`: immutable route lineage plus the current head Checkpoint;
- `workflow_checkpoints`: append-only versioned snapshots with parent references, Stage key, snapshot JSON, and creation reason;
- `route_stage_projections`: one manifest Stage projection per Route;
- `workflow_threads`: independent Stage discussions with active/archived status.

Context creation is one transaction: Context, main Route, initial Stage projections, and a version-zero bootstrap Checkpoint are inserted together. Route branching is also one transaction: the source Checkpoint is locked and ownership-checked, a new Route is inserted, its Stage projections are restored from the source snapshot, and a version-zero branch Checkpoint becomes its head.

Checkpoint immutability is enforced both by repository interfaces and a database trigger. Route-head foreign-key validation is enforced by a deferred constraint trigger because Route and Checkpoint creation is cyclic inside one transaction.

## 5. Snapshot contract

Each Checkpoint stores a stable JSON object:

```json
{
  "stages": [
    {
      "stage_key": "discover",
      "status": "active",
      "internal_state": "start"
    }
  ]
}
```

Stage status is one of `not_started`, `active`, or `completed`. Phase 3 creates the first manifest Stage as `active` and all later Stages as `not_started`. The snapshot format is intentionally sufficient for persistence and branching without pretending that workflow execution exists.

## 6. API

All domain endpoints require the existing HttpOnly session and same-origin checks for mutations.

- `GET /api/contexts` lists the current user's Contexts.
- `POST /api/contexts` creates a Context and its main Route.
- `GET /api/contexts/:contextId/workspace` returns Context metadata and Routes.
- `GET /api/routes/:routeId/workspace?stage=<key>&checkpoint=<uuid>` returns the selected Route, manifest Stage projections, Threads for the selected Stage, Checkpoints, and selected/head Checkpoint metadata.
- `POST /api/routes/:routeId/threads` creates a Thread for a manifest Stage.
- `PATCH /api/threads/:threadId` renames or archives a Thread.
- `POST /api/contexts/:contextId/routes` creates a Route from a visible historical Checkpoint.

Validation errors use `400 INVALID_REQUEST`; missing or cross-user resources use `404 NOT_FOUND`; duplicate command-independent conflicts use `409 CONFLICT`; missing sessions use `401 UNAUTHENTICATED`.

## 7. Web experience

After login, the application loads `/api/contexts`.

- With no Context, it shows a focused creation screen.
- The workspace URL is `/contexts/:contextId/routes/:routeId/stages/:stageKey` with optional `checkpoint` and `thread` query parameters.
- The left navigation provides Context switching, Route switching, Stage navigation, and a Checkpoint timeline.
- The central workspace shows the selected Stage and whether the user is browsing the Route head or history. Historical browsing exposes an explicit “从此检查点创建新路线” action.
- The right panel lists persisted Threads for the selected Stage and allows creation, selection, renaming, and archiving.
- Draft storage remains browser-local and is keyed by the complete workspace path, so authentication expiry does not erase unsent text.
- Mobile uses the same information hierarchy in a single-column layout.

No control claims to send a workflow message, advance a Stage, or adopt a conclusion during Phase 3.

## 8. Security and failure handling

The domain service receives an authenticated user ID rather than trusting user IDs from request bodies. Stage keys are checked against the private parsed manifest. Titles are trimmed and limited to 1-120 characters. All mutations require the existing origin guard.

Unexpected domain failures are sanitized as `503 DOMAIN_SERVICE_UNAVAILABLE`; internal stack traces are not returned. The identity error contract remains unchanged.

## 9. Verification

Acceptance requires fresh evidence for:

1. migration idempotency and checkpoint database immutability;
2. Context bootstrap transaction and Stage projections;
3. Thread isolation by user, Route, and Stage;
4. historical Route branching without source mutation;
5. HTTP authentication, origin, validation, and cross-user isolation;
6. React empty state, Stage navigation, Thread creation, Checkpoint browsing, and Route branching;
7. production Docker + PostgreSQL + Mailpit + Playwright journey;
8. persistence across Web-container force-recreation;
9. desktop and 390-pixel layouts;
10. no LibreChat/Mongo runtime and no public PostgreSQL port;
11. native opt-in and legacy-default preservation.

The production browser journey is:

```text
register -> verify -> login -> create Context -> browse Stage
-> create two Threads -> select historical Checkpoint -> branch Route
-> refresh -> force-recreate Web -> restore Context/Route/Stage/Thread
```

## 10. Follow-on boundary

Phase 4 adds the Workflow Bridge, command idempotency, messages, SSE, controlled named actions, adoption, and workflow-generated Checkpoints. Phase 3 must not create placeholder implementations of those capabilities.
